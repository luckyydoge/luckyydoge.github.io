Operating system interfaces

介绍了操作系统的基础概念和常用接口。

Processes and memory

进程是运行在内存中的程序，要想创建新进程可以通过调用fork()接口。操作系统是第一个进程，在开机时通过写死在磁盘的程序创建，其他所有进程都是操作系统通过fork()等接口创建的。

I/O and FIle descriptors

文件描述符是一个integer，每个进程都有一张自己维护的文件描述符表，kernel有所有进程的文件描述符。

文件描述符对底层硬件和概念进行了抽象，简化了I/O操作。

文件描述符中的0、1、2在进程刚创建时就已经分配,分别是标准输入、标准输出、标准错误输出。

Pipes

管道是一种特殊的文件描述符，用于进程间的通信。

管道在内核中拥有内存，对管道进行写入和读出都是对内核中的缓存进行操作。

由于管道也是文件描述符，因此可以共用接口。

File system

文件系统将文件分成了三个种类：file, directory, device。这三种文件的底层都是inode。

Operating system organization

介绍操作系统的组成

Abstracting physical resources

操作系统的一大功能是抽象物理设备，为其他软件提供服务。当软件想使用硬件服务时，不需要自己编写驱动，只需调用操作系统提供的接口即可。这样方便了软件的开发。

另一大功能是实现了资源的共享，让软件可以共同使用硬件资源。操作系统同时是资源的分配者，根据每个软件的需要为其分配资源。

操作系统也实现了软件之间的隔离。在软件看来，自己是享有整个硬件资源的，不同的进程并不能相互看见。因此也保障了进程的安全，防止恶意软件入侵。

User mode,supervisor mode,and system calls

普通进程和操作系统运行在不同的模式下，操作系统的权限比普通进程高，因此当普通进程出现错误时，操作系统可以清理该进程，释放其享有的资源。

普通进程想调用硬件资源时，必须经过内核。进程使用system call告诉内核自己的需求，内核处理完需求再返回给进程。

Kernel organization

内核有两种设计理念：微内核和宏内核。

微内核中只有必要的部分是处于supervisor模式，其余部分则是user模式。这种设计的好处是提升了系统的可靠性，缺点是降低性能和实现复杂。

宏内核则是整个内核都处于supervisor模式。其优缺点与微内核刚好相反。

Code: xv6 organization

kernel/defs.h中展示了内核中的所有函数。

Process overview

每个进程都有一张自己的页表，在进程看来自己享有整个内存空间，这个内存空间称为虚拟内存，页表的作用是将虚拟内存翻译成物理地址。

进程内存空间由低到高分别是：代码和全局变量、栈、堆、trapframe、trampoline。

Lab Net：Network driver

实现两个函数：e1000_transmit() 和 e1000_recv(), 作用分别是发送包和接受包。

当网络栈需要发送数据时，调用e1000_transmit()，用mbuf作为参数，其中包含了需要发送的数据的地址。transmit()函数需要将这个地址放入TXring中，这是一个缓存区，存放将要发送的数据。transmit()要在发送完成时释放被发送的数据占有的资源（此时E1000_TX_STAT_DD被设置为1）。

当E1000接收数据时，会将数据直接存放在预先分配给RX中的内存，然后调用中断。e1000_recv()需要处理中断，扫描RXring，将所有新接收的包放入网络栈，为每个使用过的RX分配新资源。

这个实验说要看400页的文档，其实只要把3.2.6 Receive Ring 和3.4 Transmit Ring 看完，了解这两个结构是如何工作就行。还有就是3.3中CMD的部分。

recv()函数中如果设置了锁反而通过不了测试，通过打印调试发现是在进入net_rx()函数时出现了锁竞争。推测是执行net_rx()时系统又接收到数据发生了中断，此时recv()函数没有执行完，锁仍在保持状态，导致锁竞争。去除锁后由于有检查E1000_RX_STAT_DD位保证不会出现数据竞争，因此能正常运行。

注意recv()函数中一次中断要将所有接收的数据处理掉。

Lab:locks

Memory allocator (moderate)

由于xv6只有一个内存管理表，却有多个CPU，因此容易出现多个CPU同时申请内存，此时由于只有一个表，会出现竞争。

实验目的是实现每个cpu都有一个自己的内存管理表，由于每个CPU在同一时间只会处理一个任务，因此可以大大减少竞争。

可能会出现一个CPU的内存表已经用完而其他CPU还有空闲，此时不因发生内存已满的错误，而是从还有空闲的CPU那里借用空间。

Buffer cache

由于xv6的缓存区只有一个锁，在高强度读写时会出现大量锁竞争，实验要求实现多个锁来减少锁竞争。

取消双向链表记录最近使用的块，改用时间戳记录。在buf的结构中添加time记录释放的时间。

以每个buf的blockno为key，使用哈希表存储这些buf，方便查找。每个bucket都有一个自己的锁。

初始化的时候将所有buf循环放进每个bucket，防止遗漏。

调用bget()时，先查找blockno对应的id有无存在的buf有则返回。若无，则先遍历当前的桶，寻找有无可用的块。若无，则遍历整个缓存，寻找最少使用的未用块。

遍历当前桶时获取当前桶的锁，遍历整个缓存时先释放桶级锁，再获取表级锁，防止死锁。（当A想获取表级锁时，此时A是已经持有A的锁，如果同时B想获取表级锁，B必须等待A遍历完才能获得表级锁，而A遍历可能会尝试获取B锁，这样就形成了死锁。如果在获取表级锁之前就将自己持有的锁释放，就能避免这种死锁）

brelease()在rec为0时，在buf的time打上时间戳。

bpin()和bunpin()则是将原来的表级锁换成桶级锁

Lab：File System

Large files

实现文件系统的扩展，将单个文件的最大大小从268KB扩展到65803KB。

原有文件存储方式是12个直接链接+一个间接链接，间接链接指向一个存储了256个地址的块，每个地址都是一个可以存储文件的块，因此文件最大为256 + 12 = 268KB。

新的文件结构使用双重间接链接，11块直接链接，一个块间接链接（提供256个块），一个块双重间接链接。（指向一个a块，a块内存储256个地址，每个地址又指向一个b块，b块中也存储256个地址，b块中的地址指向的才是存储文件内容的块）

Symbolic links

实现软连接 symlink(char *target, char *path),其中target是链接指向的目标，path则是要创建链接的路径。

找到path的父目录，在其父目录下创建名为path的inode，内容是target。

系统调用open()需要能够识别symlink类文件，在打开此类文件时打开指向的文件。当目标文件不存在是返回错误代码。

为防止指向自身导致的死循环，设置指向symlink的深度最多为10。

当操作是O_NOFOLLOW时，并不寻找指向的文件。
